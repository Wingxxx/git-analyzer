## 概述

### 功能说明

贡献度分析通过多维度指标量化评估开发者在Git仓库中的贡献情况，生成客观的贡献度排名。

### 使用场景

- 团队绩效评估
- 项目贡献者识别
- 代码审查资源分配
- 开源项目贡献统计

### 命令格式

```bash
# 使用新算法（默认）
python scripts/git_analyzer.py analyze [path] [--since YYYY-MM-DD] [--until YYYY-MM-DD]

# 使用旧算法
python scripts/git_analyzer.py analyze [path] --algorithm legacy [--since YYYY-MM-DD] [--until YYYY-MM-DD]
```

### 参数说明

| 参数 | 类型 | 必需 | 默认值 | 说明 |
|------|------|------|--------|------|
| path | string | 否 | 当前目录 | 仓库路径 |
| --since | date | 否 | - | 开始日期 |
| --until | date | 否 | - | 结束日期 |
| --algorithm | string | 否 | quality | 算法模式：legacy(旧算法) 或 quality(新算法) |

## 算法模式

### 旧算法（LEGACY模式）

**核心思想**: 基于数量统计的贡献度评估

**计算公式**:
```
总得分 = 提交次数得分 + 文件变更得分 + 代码变更得分 + 价值得分
```

**适用场景**: 快速统计、历史数据对比

### 新算法（QUALITY模式，默认）

**核心思想**: 基于代码质量的综合贡献度评估

**计算公式**:
```
总得分 = 代码质量得分 × 0.6 + 协作贡献得分 × 0.25 + 文档贡献得分 × 0.15
```

**适用场景**: 绩效评估、团队协作分析、代码质量改进

---

## 新算法评分指标

### 1. 代码质量得分（权重60%）

基于五维度代码质量评估：

#### 1.1 可读性评估（0-25分）

- 命名规范（0-5分）
- 注释质量（0-5分）
- 代码格式（0-5分）
- 代码复杂度（0-5分）
- 文档字符串（0-5分）

#### 1.2 可维护性评估（0-25分）

- 模块化程度（0-5分）
- 代码重复度（0-5分）
- 依赖管理（0-5分）
- 函数长度（0-5分）
- 类设计（0-5分）

#### 1.3 性能效率评估（0-20分）

- 算法复杂度（0-5分）
- 资源使用（0-5分）
- 数据结构选择（0-5分）
- 循环优化（0-5分）

#### 1.4 错误处理评估（0-15分）

- 异常捕获（0-5分）
- 错误日志（0-5分）
- 边界检查（0-5分）

#### 1.5 安全性评估（0-15分）

- 输入验证（0-5分）
- 敏感数据处理（0-5分）
- 安全编码实践（0-5分）

### 2. 协作贡献得分（权重25%）

#### 2.1 代码审查贡献

识别提交消息中的关键词：
- review, reviewed, approve, approved, lgtm, looks good

**得分**: 每次审查 +2.0分

#### 2.2 问题解决贡献

识别提交消息中的关键词：
- fix, bug, issue, resolve, close, closed, closes

**得分**: 
- 基础得分 +2.0分
- 包含Issue编号引用（#数字）额外 +1.0分

#### 2.3 合并贡献

识别提交消息中的关键词：
- merge, merged, merge pull, merge branch

**得分**: 每次合并 +1.5分

### 3. 文档贡献得分（权重15%）

#### 3.1 文档完整性得分

**计算方式**: 每个文档文件 +2.0分（最多10分）

**文档文件扩展名**: .md, .txt, .rst, .adoc, .tex

#### 3.2 文档质量得分

**计算方式**:
- 新增文档内容：净增行数 × 0.1（最多10分）
- 删除文档内容：绝对值 × 0.05（最多5分）

## 计算算法

### 新算法计算流程

```
1. 遍历所有提交记录
2. 对每个提交：
   a. 提取代码文件、文档文件
   b. 评估代码质量（五维度评估）
   c. 分析协作贡献（审查、问题解决、合并）
   d. 分析文档贡献（数量、质量）
3. 按作者汇总数据
4. 计算每位作者的各项得分
5. 加权计算总得分
6. 按总得分排序
```

### 权重配置

```python
QUALITY_WEIGHT = 0.6        # 代码质量权重
COLLABORATION_WEIGHT = 0.25  # 协作贡献权重
DOCUMENTATION_WEIGHT = 0.15  # 文档贡献权重
```

## 注意事项

### 新算法优势

1. **防止刷分**: 移除易刷分指标
2. **质量导向**: 真正评估代码质量
3. **协作重视**: 认可团队协作贡献
4. **文档认可**: 公平对待文档贡献者

### 评分局限性

1. **代码审查**: 仅基于提交消息识别
2. **Issue处理**: 需要在提交消息中引用Issue编号
3. **文档质量**: 基于行数统计，未评估内容质量

### 建议使用方式

1. **作为参考指标**: 不作为唯一评估标准
2. **结合定性分析**: 综合代码质量、团队协作等因素
3. **定期回顾**: 定期分析趋势而非单点数据
4. **使用新算法**: 推荐使用基于代码质量的新算法

## 最佳实践

### 1. 选择合适的算法模式

#### 使用新算法（QUALITY模式）的场景

- **绩效评估**：需要公平评估团队成员的实际贡献
- **代码质量改进**：关注代码质量而非数量
- **团队协作分析**：重视代码审查、问题解决等协作行为
- **文档贡献认可**：需要公平对待文档贡献者

#### 使用旧算法（LEGACY模式）的场景

- **快速统计**：需要快速获取贡献数量统计
- **历史数据对比**：与历史数据保持一致性
- **迁移过渡期**：从旧算法向新算法过渡时进行对比

### 2. 提高贡献度评分的技巧

#### 提高代码质量得分

| 维度 | 最佳实践 | 评分提升 |
|------|---------|---------|
| 可读性 | 使用有意义的变量名、添加必要注释、保持代码格式统一 | +5~15分 |
| 可维护性 | 模块化设计、减少代码重复、合理拆分函数 | +5~15分 |
| 性能效率 | 选择合适的数据结构、优化循环、减少资源消耗 | +5~10分 |
| 错误处理 | 完善异常捕获、添加边界检查、记录错误日志 | +5~10分 |
| 安全性 | 输入验证、敏感数据加密、遵循安全编码规范 | +5~10分 |

#### 提高协作贡献得分

**代码审查贡献**：
- 在提交消息中使用关键词：`review`, `reviewed`, `approve`, `approved`, `lgtm`, `looks good`
- 示例：`Reviewed and approved PR #123`

**问题解决贡献**：
- 在提交消息中使用关键词：`fix`, `bug`, `issue`, `resolve`, `close`, `closes`
- 引用Issue编号获得额外加分
- 示例：`Fix login bug, closes #456`

**合并贡献**：
- 在提交消息中使用关键词：`merge`, `merged`, `merge pull`, `merge branch`
- 示例：`Merge pull request #789 from feature/new-api`

#### 提高文档贡献得分

- 编写完善的项目文档（README、API文档、用户指南）
- 更新过时的文档内容
- 添加代码示例和教程
- 文档文件扩展名：`.md`, `.txt`, `.rst`, `.adoc`, `.tex`

### 3. 提交消息规范

推荐使用规范的提交消息格式：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型（type）**：
- `feat`: 新功能
- `fix`: 修复bug
- `docs`: 文档更新
- `style`: 代码格式调整
- `refactor`: 代码重构
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**：
```
feat(auth): add OAuth2 authentication

- Implement OAuth2 login flow
- Add token refresh mechanism
- Update user session management

Closes #123
Reviewed-by: John Doe
```

### 4. 团队协作建议

1. **定期分析**：建议每月或每季度进行一次贡献度分析
2. **多维度评估**：结合代码质量、协作贡献、文档贡献综合评估
3. **趋势分析**：关注贡献度变化趋势而非单点数据
4. **公平对待**：认可不同类型的贡献（代码、文档、协作）

## 常见问题解答

### Q1: 新算法和旧算法有什么区别？

**主要区别**：

| 特性 | 新算法（QUALITY） | 旧算法（LEGACY） |
|------|------------------|-----------------|
| 核心思想 | 基于代码质量评估 | 基于数量统计 |
| 防刷分 | 有效防止刷分 | 容易被刷分 |
| 质量导向 | 高质量代码得分高 | 数量多得分高 |
| 协作认可 | 重视协作贡献 | 不评估协作 |
| 文档认可 | 公平对待文档贡献 | 文档贡献权重低 |
| 适用场景 | 绩效评估、质量改进 | 快速统计、历史对比 |

### Q2: 如何提高代码质量得分？

**建议**：
1. **可读性**：使用有意义的命名、添加注释、保持格式统一
2. **可维护性**：模块化设计、减少重复代码、合理拆分函数
3. **性能效率**：选择合适的数据结构、优化算法复杂度
4. **错误处理**：完善异常捕获、添加边界检查
5. **安全性**：输入验证、敏感数据加密

### Q3: 协作贡献得分如何计算？

**计算方式**：
- **代码审查**：提交消息包含 `review`, `approve`, `lgtm` 等关键词，每次 +2.0分
- **问题解决**：提交消息包含 `fix`, `bug`, `issue` 等关键词，基础 +2.0分，引用Issue编号额外 +1.0分
- **合并贡献**：提交消息包含 `merge` 等关键词，每次 +1.5分

### Q4: 文档贡献得分如何计算？

**计算方式**：
- **文档完整性**：每个文档文件 +2.0分（最多10分）
- **文档质量**：新增文档内容每行 +0.1分（最多10分），删除文档内容每行 +0.05分（最多5分）
- **文档文件类型**：`.md`, `.txt`, `.rst`, `.adoc`, `.tex`

### Q5: 为什么推荐使用新算法？

**优势**：
1. **公平性**：防止通过大量低质量提交刷分
2. **质量导向**：鼓励编写高质量代码
3. **协作重视**：认可代码审查、问题解决等协作行为
4. **文档认可**：公平对待文档贡献者

### Q6: 如何在团队中推广新算法？

**建议步骤**：
1. **对比分析**：同时运行新旧算法，展示差异
2. **培训说明**：向团队解释新算法的评分机制
3. **制定规范**：建立提交消息规范和代码质量标准
4. **定期回顾**：每月回顾贡献度分析结果，持续改进

### Q7: 分析结果不准确怎么办？

**可能原因和解决方案**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 协作贡献为0 | 提交消息未使用关键词 | 规范提交消息格式 |
| 文档贡献为0 | 未提交文档文件 | 添加项目文档 |
| 代码质量得分低 | 代码质量问题 | 参考代码质量评估指南改进 |
| 总得分异常 | 时间范围设置不当 | 检查--since和--until参数 |

### Q8: 如何查看详细的代码质量评估？

**方法**：
1. 查看 `references/quality_evaluation.md` 了解评估标准
2. 使用 `report` 命令生成详细报告
3. 查看报告中的代码质量得分详情
